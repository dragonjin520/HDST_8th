# 전사 회의실 예약 시스템

## 1. 전체 시스템 범위

전사 임직원이 회의실을 조회하고 예약·변경·취소하며, 관리자가 회의실과 운영 상태를 관리하는 시스템이다.

---
## 전사 회의실 예약 시나리오

1. 직원이 예약 시스템에 로그인한다.
2. 원하는 회의실을 선택한다. (주소, 빌딩, 층, 호수)

    * 비품, 최대인원수 확인

1. 원하는 시간을 선택한다. (년, 월, 일, 시, 분)

    * 예약 가능한 시간대 확인
        * 시간을 먼저 확정할지 회의실을 먼저 확정할지?
            * 특정날짜에 꼭 시행되어야 하지만 회의실은 상관없는 경우
            * 회의실이 먼저 정해진 경우

1. 예약 목적, 참여인원, 예약자명(이름,부서) 작성
2. 예약 확정
3. 예약 내역 확인
4. 예약 변경/취소
    1. 현재 예약 사항과 겹치는 문제?
5. 회의실 이용




## 2. 주제영역 정의

### 2.1 조직

**목적:** 회의실 예약 시스템을 이용하는 회사, 부서, 직원, 역할 및 접근 권한 정보를 관리한다.

#### 요구사항

1. 하나의 회사는 여러 부서를 가질 수 있다.
2. 하나의 부서는 하나의 회사에 속한다.
3. 하나의 부서는 여러 직원을 가질 수 있다.
4. 하나의 직원은 하나의 부서에 속한다.
5. 하나의 직원은 하나의 권한등급을 가진다.
6. 하나의 권한등급은 여러 직원에게 부여될 수 있다.
7. 직원은 재직, 휴직, 퇴직 등의 재직 상태를 가진다.
8. 퇴직한 직원은 새로운 예약을 생성할 수 없다.
9. 퇴직한 직원의 미래 예약은 예약 영역에서 취소 처리한다.
10. 직원이 퇴직하더라도 기존 예약 이력 보존을 위해 직원 정보는 삭제하지 않는다.

#### 후보 엔터티 선정 규칙

1. 가능성이 있으면 우선 후보에 올린다.
2. 깊은 구조보다 후보 자격만 확인한다.
3. 동의어를 바로 제거하지 않는다.
4. 모호한 개념은 상식적인 의미로 임시 정의한다.
5. 프로세스와 엔터티를 혼동하지 않는다.
6. 예외 상황 때문에 후보 도출을 멈추지 않는다.

#### 후보 엔터티

- 회사
- 부서
- 직원
- 권한등급
- 역할
- 재직상태
- 관리자

#### 확정 엔터티 및 속성

##### 회사
- 회사명: 주식별자

##### 부서
- 부서ID: 주식별자
- 부서명: 선택 속성
- 회사명: 회사 엔터티를 참조하는 외래식별자

##### 직원
- 직원ID: 주식별자
- 이름: 필수 속성
- 휴대전화번호: 필수 속성
- 입사일: 필수 속성
- 부서ID: 부서 엔터티를 참조하는 외래식별자
- 권한등급: 접근권한 엔터티를 참조하는 외래식별자
- 사내전화번호: 선택 속성
- 이메일: 선택 속성
- 재직상태: 필수 속성
- 휴대전화번호: 개인정보

##### 권한등급

- 권한등급: 주식별자
- 권한명: 선택 속성

접근권한은 숫자형 등급으로 관리하며, 직원의 보유 권한 등급과 회의실의 최소 권한 등급을 비교하여 예약 가능 여부를 판단한다.

```text
직원 보유 권한 등급 >= 회의실 최소 권한 등급
```

#### 관계 정의

```text
회사 1 ── N 부서
부서 1 ── N 직원
권한등급 1 ── N 직원
```

- 하나의 회사는 여러 부서를 가질 수 있다.
- 하나의 부서는 반드시 하나의 회사에 속한다.
- 하나의 부서는 여러 직원을 가질 수 있다.
- 하나의 직원은 반드시 하나의 부서에 속한다.
- 하나의 접근권한은 여러 직원에게 부여될 수 있다.
- 하나의 직원은 하나의 권한등급을 가진다.
- 하나의 회의실은 하나의 최소권한등급을 가진다.
- 직원은 자신의 권한등급보다 높은 최소권한등급을 가진 회의실을 예약/취소할 수 없다.
- 하나의 직원은 하나 이상의 관리내역을 가질 수 있다.
- 하나의 관리내역은 반드시 하나 이상의 직원을 가진다.


- 하나의 회의실은 반드시 하나 이상의 관리내역을 가진다?


- 참조자는 자신의 권한등급보다 높은 최소권한등급을 가진 회의실을 변경할 수 있다.


#### 서브타입 검토

현재 조직 영역에서는 명확한 서브타입을 두지 않는다.

- 관리자: 직원의 본질적인 유형이 아니라 시스템 역할로 본다.
- 재직·휴직·퇴직: 직원의 상태값으로 본다.
- 임원·R&D·일반: 직원 서브타입이 아니라 권한 등급으로 본다.

### 2.2 자원

**목적:** 예약 대상이 되는 건물과 회의실, 회의실 이용에 필요한 자원 정보를 관리한다.

#### 업무 질문

- 어느 회사와 건물에 있는 회의실인가?
- 회의실은 몇 층, 몇 호에 위치하는가?
- 몇 명까지 사용할 수 있는가?
- 현재 예약 가능한 상태인가?
- 어떤 권한 등급의 직원이 예약할 수 있는가?
- 회의실 관리자는 누구인가?

#### 현재 ERD 기준 엔터티 및 속성

##### 건물

- 건물ID: 주식별자
- 주소: 필수 속성
- 회사명: 회사 엔터티를 참조하는 외래식별자

##### 회의실

- 회의실ID: 주식별자
- 건물ID: 건물 엔터티를 참조하는 외래식별자
- 방호수: 필수 속성
- 층: 필수 속성
- 최소권한등급: 접근권한 엔터티를 참조하는 외래식별자
- 관리자직원ID: 직원 엔터티를 참조하는 외래식별자
- 수용인원: 선택 속성

#### 관계 정의

```text
회사 1 ── N 건물
건물 1 ── N 회의실
권한등급 1 ── N 회의실
직원 1 ── N 회의실 : 관리한다
```

- 하나의 회사는 여러 건물을 가질 수 있다.
- 하나의 건물은 반드시 하나의 회사에 속한다.
- 하나의 건물은 여러 회의실을 가질 수 있다.
- 하나의 회의실은 반드시 하나의 건물에 속한다.
- 하나의 권한등급은 여러 회의실의 최소권한등급으로 설정될 수 있다.
- 하나의 회의실은 하나의 상태를 가진다.


- 하나의 직원은 여러 회의실을 관리할 수 있다.

### 2.3 예약

**목적:** 직원이 회의실을 일정 시간 동안 사용하기 위한 신청, 변경, 취소 및 대리 관리 정보를 관리한다.

#### 업무 질문

- 누가 어떤 회의실을 예약했는가?
- 언제부터 언제까지 사용하는가?
- 현재 예약 상태는 무엇인가?
- 예약자는 누구인가?
- 예약자를 대신하여 변경·취소할 수 있는 참조자는 누구인가?
- 예약이 변경되거나 취소되었는가?

#### 엔터티 및 속성

##### 예약

- 예약ID: 주식별자
- 참여인원: 필수 속성
- 회의시작시간: 필수 속성
- 회의종료시간: 필수 속성
- 예약자ID: 직원 엔터티를 참조하는 외래식별자
- 회의실ID: 회의실 엔터티를 참조하는 외래식별자
- 예약상태코드: 필수 속성
- 참조자ID: 직원 엔터티를 참조하는 선택 외래식별자
- 회의목적: 선택 속성

#### 참조자 정의

참조자는 예약자가 피치 못할 사정으로 예약을 직접 관리할 수 없을 때, 해당 예약을 대신 변경하거나 취소할 수 있도록 지정된 직원이다.

#### 참조자 업무 규칙

1. 하나의 예약은 반드시 한 명의 예약자를 가진다.
2. 하나의 예약은 한 명 이상의 참조자를 가질 수 있다.
3. 참조자는 직원 중에서 지정한다.
    - 참조자는 여러 명일 수 있다.
4. 참조자는 해당 예약을 변경하거나 취소할 수 있다.
5. 예약자와 참조자는 서로 다른 직원이어야 한다.
6. 재직상태가 퇴직인 직원은 참조자로 지정할 수 없다.

#### 관계 정의

```text
직원 1 ── N 예약 : 예약한다
직원 1 ── 0..N 예약 : 참조자로 지정된다
회의실 1 ── N 예약
```

- 하나의 직원은 여러 예약을 생성할 수 있다.
- 하나의 예약은 반드시 한 명의 예약자를 가진다.
- 하나의 직원은 여러 예약의 참조자가 될 수 있다.
- 하나의 예약은 참조자를 지정하지 않을 수도 있다.
- 하나의 회의실은 같은 일자에 여러 예약을 가질 수 있다. - 같은 날 다른 시각
- 하나의 예약은 반드시 하나의 회의실을 대상으로 한다.

#### 예약 핵심 업무 규칙

1. 직원의 보유 권한 등급이 회의실의 최소 권한 등급 이상이어야 예약할 수 있다.
2. 동일한 회의실은 시간이 겹치는 둘 이상의 활성 예약을 가질 수 없다.
3. 회의 시작 시간은 회의 종료 시간보다 빨라야 한다.
4. 예약 상태는 예약 확정, 변경, 취소의 상태를 구분할 수 있어야 한다.
5. 취소된 예약은 삭제하지 않고 상태를 변경하여 보존한다.

## 4. 추후 검토 사항

직원
- 관리자 역할을 별도의 역할 엔터티로 분리할지 검토
    - 관리자 변경 이력을 관리하기 위해 분리



회의실
- 회의실의 종류를 구분할 것인지?
    - 화상회의실
    - 권장 인원수 기준
    - 비품 구비 기준
- 회사명 대신 별도의 회사ID를 주식별자로 사용할지 검토
- 회사 대표자와 부서장을 직원 외래식별자로 관리할지 검토
- 회의실명과 건물명을 별도 속성으로 추가할지 검토

예약
- 회의 시작·종료 시간을 날짜와 시간이 결합된 일시 형식으로 관리할지 검토
    * 예약 날짜 정보 단위 설정
        * 주별 예약? (7월 3주차 수요일)
        * 일별 예약?  (0722)
- 예약 변경 이력 엔터티 추가 여부 검토
- 반복 예약 기능과 반복 예약 충돌 처리 방식 검토
* 언제 전까지 취소 가능하게 할건지?
* 업무시간 외에 예약가능한지?
    * 24시간 가능한지?
* 취소된 예약은 삭제하지 않고 상태를 변경하여 보존할것인지?
    - 상태를 update하는지 변경된 예약을 신규 저장하는지?



* 어느 주기로 직원 리스트 체크할건지?
* 회의가 일찍 끝났지만 예약이 계속 유지됨
    * 회의가 끝났다는 것을 어떻게 알것인지?
