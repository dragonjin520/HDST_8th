# 전사 회의실 예약 시스템

## 1. 전체 시스템 범위

전사 임직원이 회의실을 조회하고 예약·변경·취소하며, 관리자가 회의실과 운영 상태를 관리하는 시스템이다.

---
## 전사 회의실 예약 시나리오

1. 직원이 예약 시스템에 로그인한다.
2. 원하는 회의실을 검색한다.
    - 주소, 빌딩, 층, 회의실명
2. 회의실을 선택한다.
    - 장비, 수용인원 확인
3. 예약을 생성한다. (년, 월, 일 / 시, 분)
    - 예약 가능한 시간 확인
    - 예약 시작 시각 선택
    - 예약 종료 시각 선택
4. 참여자 작성
5. 요구사항 작성
6. 예약 확정
7. 예약 내역 확인
8. 예약 변경/취소


### 직원

#### 요구사항

1. 하나의 법인은 여러 부서를 가질 수 있다.
2. 하나의 부서는 하나의 법인에 속한다.
3. 하나의 부서는 여러 직원을 가질 수 있다.
4. 하나의 직원은 하나의 부서에 속한다.
5. 하나의 직원은 하나의 권한등급을 가진다.
6. 하나의 권한등급은 여러 직원에게 부여될 수 있다.
7. 직원은 재직, 휴직, 퇴직의 재직 상태를 가진다.
8. 퇴직한 직원의 예약은 예약 이력에서 예약 상태를 취소(코드)로 기록한다.
9. 직원이 퇴직하더라도 기존 예약 이력 보존을 위해 직원 정보는 삭제하지 않는다.


#### 확정 엔터티 및 속성

##### 법인
- 법인명: 주식별자

##### 부서
- 부서ID: 주식별자
- 부서명: 선택 속성
- 법인명: 법인 엔터티를 참조하는 외래식별자

##### 직원
- 직원ID: 주식별자
- 직원명: 필수 속성
- 전화번호: 필수 속성
- 입사일: 필수 속성
- 부서ID: 부서 엔터티를 참조하는 외래식별자
- 권한등급: 접근권한 엔터티를 참조하는 외래식별자
- 사내전화번호: 선택 속성
- 이메일: 선택 속성
- 재직상태: 필수 속성
- 전화번호: 개인정보

##### 권한등급

- 권한등급: 주식별자
- 권한명: 선택 속성

접근권한은 숫자형 등급으로 관리하며, 직원의 보유 권한 등급과 회의실의 최소 권한 등급을 비교하여 예약 가능 여부를 판단한다.

##### 직원상태
- 직원ID
- 재직상태
- 시작날짜
- 종료날짜
    - 종료날짜 NULL값이 최신값

##### 직원상태이력
- 직원ID

#### 관계 정의

```text
법인 1 ── N 부서
부서 1 ── N 직원
권한등급 1 ── N 직원
```

- 하나의 법인은 여러 부서를 가질 수 있다.
- 하나의 부서는 반드시 하나의 법인에 속한다.
- 하나의 부서는 여러 직원을 가질 수 있다.
- 하나의 직원은 반드시 하나의 부서에 속한다.
- 하나의 접근권한은 여러 직원에게 부여될 수 있다.
- 하나의 직원은 반드시 하나의 권한등급을 가진다.
- 하나의 회의실은 반드시 하나의 최소권한등급을 가진다.
- 하나의 직원은 반드시 하나의 직원상태를 가진다.
- 하나의 직원상태는 반드시 하나의 재직상태를 가진다. 
- 직원은 자신의 권한등급보다 높은 최소권한등급을 가진 회의실을 예약/취소할 수 없다.


- 재직상태가 퇴직인 직원의 예약은 예약 이력에서 예약상태를 취소로 기록한다.
- 직원의 재직상태가 퇴직으로 변경되어도 기존 예약이력 보존을 위해 직원 정보는 삭제하지 않는다.

관리내역? 관리자?
- 하나의 직원은 하나 이상의 관리내역을 가질 수 있다.
- 하나의 관리내역은 반드시 하나 이상의 직원을 가진다.
- 하나의 회의실은 반드시 하나 이상의 관리내역을 가진다?


- 대리인은 자신의 권한등급보다 높은 최소권한등급을 가진 회의실을 변경할 수 있다.

## 회의실

#### 엔터티 및 속성

##### 건물

- 건물ID: 주식별자
- 주소: 필수 속성
- 법인명: 법인 엔터티를 참조하는 외래식별자

##### 회의실

- 회의실ID: 주식별자
- 건물ID: 건물 엔터티를 참조하는 외래식별자
- 방호수: 필수 속성
- 층: 필수 속성
- 최소권한등급: 접근권한 엔터티를 참조하는 외래식별자
- 관리자ID: 직원 엔터티를 참조하는 외래식별자
- 수용인원: 선택 속성

#### 관계 정의

```text
법인 1 ── N 건물
건물 1 ── N 회의실
권한등급 1 ── N 회의실
직원 1 ── N 회의실 : 관리한다
```

- 하나의 법인은 여러 건물을 가질 수 있다.
- 하나의 건물은 반드시 하나의 법인에 속한다.
- 하나의 건물은 여러 회의실을 가질 수 있다.
- 하나의 회의실은 반드시 하나의 건물에 속한다.
- 하나의 권한등급은 여러 회의실의 최소권한등급으로 설정될 수 있다.
- 하나의 회의실은 하나의 상태를 가진다.
- 하나의 직원은 여러 회의실의 관리자가 될 수 있다.

### 2.3 예약

#### 엔터티 및 속성

##### 예약

- 예약ID: 주식별자
- 참여인원: 필수 속성
- 회의시작시간: 필수 속성
- 회의종료시간: 필수 속성
- 예약자ID: 직원 엔터티를 참조하는 외래식별자
- 회의실ID: 회의실 엔터티를 참조하는 외래식별자
- 예약상태코드: 필수 속성
- 대리자ID: 직원 엔터티를 참조하는 선택 외래식별자
- 회의목적: 선택 속성

#### 대리인 정의

대리인는 예약자가 피치 못할 사정으로 예약을 직접 관리할 수 없을 때, 해당 예약을 대신 변경하거나 취소할 수 있도록 지정된 직원이다.

#### 대리인 업무 규칙

1. 하나의 예약은 반드시 한 명의 예약자를 가진다.
2. 하나의 예약은 한 명 이상의 대리인를 가질 수 있다.
3. 대리인는 직원 중에서 지정한다.
    - 대리인는 여러 명일 수 있다.
4. 대리인는 해당 예약을 변경하거나 취소할 수 있다.
5. 예약자와 대리인는 서로 다른 직원이어야 한다.
6. 재직상태가 퇴직인 직원은 대리인로 지정할 수 없다.

#### 관계 정의

```text
직원 1 ── N 예약 : 예약한다
직원 1 ── 0..N 예약 : 대리인로 지정된다
회의실 1 ── N 예약
```

- 하나의 직원은 여러 예약을 생성할 수 있다.
- 하나의 예약은 반드시 한 명의 예약자를 가진다.
- 하나의 직원은 여러 예약의 대리인가 될 수 있다.
- 하나의 예약은 대리인을 지정하지 않을 수도 있다.
- 하나의 회의실은 같은 일자에 여러 예약을 가질 수 있다. - 같은 날 다른 시각
- 하나의 예약은 반드시 하나의 회의실을 대상으로 한다.

#### 예약 핵심 업무 규칙

1. 직원의 보유 권한 등급이 회의실의 최소 권한 등급 이상이어야 예약할 수 있다.
2. 동일한 회의실은 시간이 겹치는 둘 이상의 활성 예약을 가질 수 없다.
3. 회의 시작 시간은 회의 종료 시간보다 빨라야 한다.
4. 예약 상태는 예약 확정, 변경, 취소의 상태를 구분할 수 있어야 한다.
5. 취소된 예약은 삭제하지 않고 상태를 변경하여 보존한다.

## 4. 추후 검토 사항


- 재직 상태 이력을 따로 둘건지?
    방식 1: 이력에서만 현재 상태 관리
    - 직원에는 현재상태를 두지 않고, 종료일시가 NULL인 이력을 현재 상태로 본다.
    - 장점은 데이터 중복이 없다는 것입니다. 단점은 현재 상태를 조회할 때 항상 이력 테이블을 확인해야 한다는 것입니다.
    
    방식 2: 직원과 이력에서 함께 관리
    - 장점은 현재 상태 조회가 편하고 빠르다는 것입니다. 단점은 두 값의 동기화를 보장해야 한다는 것입니다.

    결론 : 예약 시스템에서 재직 상태이력을 관리할 필요가 없다.
    현실에서는 재직상태이력을 인사시스템에서 관리하지만 
    이번 과제에서는 재직상태이력까지 관리하는 이유가 없다.
        재직상태이력을 조회하는 경우
            1. 퇴직 또는 휴직 시 미래 예약을 자동 취소한다.    -> ?
            2. 상태 변경 시점과 예약 취소 근거를 감사해야 한다.
            3. 과거 예약 당시 직원의 재직상태를 조회해야 한다.
            4. 인사 시스템과의 상태 변경 수신 기록을 보존해야 한다.


- 참여자와 대리인 구분 명확히
    - 대리인 -> 대리자로 변경 요청
       - 예약자와 형식을 맞추기 위해.
- 예약 내역은 누가 가능한지?
    - 아무나? 예약자만? 참여자까지?

- 법인명 대신 별도의 법인ID를 주식별자로 사용할지 검토

- 현재 예약 사항과 겹치는 문제?
    - 현재 예약을 변경하고 싶은데 지금 예약과 겹쳐서 불가능한 문제
    - 12시~1시 예약을 변경한다 -> 12시~1시 예약을 취소한다 -> 12시반~1시반 예약을 생성한다.
        - 취소한 사이에 뺏긴다? 
        - 수동으로 취소하고 생성하게 하면 번거롭다.
            - 자동으로 취소되고 생성된다면?
                - 나의 예약과 겹칠 경우 예약이 가능해야한다.
                    - 예약을 생성할때 다른 예약과 겹치는지 확인을 언제하는지?
                - 내 예약은 12시~1시 예약이라서 12시~1시 예약은 꽉차있어서 예약이 불가능하다.

* 취소된 예약은 삭제하지 않고 상태를 변경하여 보존할것인지?
    - 상태를 update하는지 변경된 예약을 신규 저장하는지?
    - overwrite를 하는것이 아니라 신규 저장해야함.

* 명 -> 동음이의어 추가 검토
    이름과 몇명 구분


* 시간을 먼저 확정할지 회의실을 먼저 확정할지?
    * 특정날짜에 꼭 시행되어야 하지만 회의실은 상관없는 경우
    * 회의실이 먼저 정해진 경우

* 로그인 시스템?
    - 사내 예약 시스템과 사내 서버가 다른지?
    - 당연히 직원이 아니면 사내 예약 시스템에 접근 불가능한지?
    - 로그인 방법?
    - 회원가입/회원탈퇴

- 퇴직한 직원은 새로운 예약을 생성할 수 없다.
    - 퇴직한 직원은 애초에 시스템에 접근이 불가능?

## 해결된 검토 사항

- 회의실의 종류를 구분할 것인지?
    - 안한다
- 관리자 역할을 별도의 역할 엔터티로 분리할지 검토
    - 관리자 변경 이력을 관리하기 위해 분리

- 회의 시작·종료 시간을 날짜와 시간이 결합된 일시 형식으로 관리할지 검토
    * 예약 날짜 정보 단위 설정
        * 년월일 / 시분초 구분해서 저장
- 예약 이력 엔터티 추가 여부 검토
    이력은 로그로 저장한다.
* 언제 전까지 취소 가능하게 할건지?
    - 1초전, 회의중에도 취소 가능함
* 업무시간 외에 예약가능한지?
    * 24시간 가능한지?
        - 8시부터 20시까지
