## 문제 정의
주어진 데이터는 택시 운전기사가 택시를 운전해서 생성해낸 데이터이다. 즉 사람이 생성한 데이터이고 사람마다, 시간마다 특성이 다를 것이다.


그런데 만약 이 데이터를 자율주행차가 만들었다면?

## Data Product 정의

Data Product는 단순히 데이터를 분석해 그래프나 통계를 만드는 것이 아니라, 수집·가공한 데이터를 이용해 특정 사용자가 반복적으로 의사결정을 내리거나 행동할 수 있도록 만든 제품 또는 기능이다.

```text
데이터 수집
→ 데이터 가공
→ 수요·비용·수익 지표 생성
→ 사용자에게 결과 제공
→ 실제 차량 배치 또는 업무 전환
```

외부 구매자에게 판매하는 경우도 있지만, 회사 내부 운영 효율화를 목적으로 사용하는 시스템 역시 Data Product에 해당한다.

이번 과제에서는 자율주행차 운영사가 차량 한 대의 가용시간당 평균 수익을 높일 수 있도록, 시간대별로 택시와 택배 업무를 전환하도록 지원하는 데이터 프로덕트를 구상했다.

## 일반 택시와 자율주행차의 차이

일반 택시와 자율주행차의 가장 큰 차이는 운전과 판단의 주체가 사람에서 시스템으로 바뀐다는 점이다.

| 구분 | 일반 택시 | 자율주행차 |
|---|---|---|
| 운전 주체 | 택시 운전사 | 자율주행 시스템 |
| 주변 환경 인식 | 운전자의 시각·청각·경험 | 카메라·라이다·레이더·GPS |
| 판단 방식 | 경험과 직관 | 알고리즘과 AI 모델 |
| 차량 배치 | 운전자가 직접 선택 | 중앙 시스템이 수요를 분석해 자동 배치 |
| 운행 가능 시간 | 운전자 근무시간에 영향 | 충전·정비 시간을 제외하면 지속 운행 가능 |
| 주요 데이터 | 승하차·거리·요금 등 운행 결과 | 운행 결과 + 센서·판단·제어·차량 상태 |

일반 택시는 주로 운행이 끝난 뒤 결과 데이터를 남긴다.

```text
언제 승객을 태웠는가
어디에서 타고 내렸는가
얼마나 이동했는가
얼마를 결제했는가
```

반면 자율주행차는 운행 결과뿐 아니라 다음과 같은 과정 데이터도 지속적으로 수집할 수 있다.

- 실시간 차량 위치
- 배터리 잔량과 예상 주행 가능 거리
- 센서 상태
- 객체 인식 결과
- 급제동·경로 변경 기록
- 자율주행 해제 및 원격 개입 기록
- 공차 이동 거리
- 현재 수행 중인 업무 유형

즉, 일반 택시 데이터가 운행 결과를 관리하기 위한 데이터라면 자율주행차 데이터는 차량의 판단과 운영을 직접 움직이는 데이터라고 볼 수 있다.

## 비즈니스 아이디어: 멀티서비스 자율주행차

자율주행 택시는 출퇴근 시간처럼 수요와 수익성이 높은 시점이 존재하지만, 그 외 시간에는 차량이 대기하면서 자원이 낭비될 수 있다.

따라서 택시 수요가 낮은 시간대를 도출한 뒤, 해당 시간 동안 택배나 배달 같은 다른 경제활동을 수행하도록 한다.

```text
07:00~11:00  택시
11:00~14:00  배달·택배
17:00~20:00  택시
02:00~05:00  허브 간 물류
```

고정된 시간표를 그대로 적용하기보다는 시간대·지역별 예상 순이익을 비교해 가장 적합한 업무를 동적으로 선택하는 방식이 더 적절하다.

### 기대 효과

- 차량 가동률 증가
- 유휴시간 감소
- 차량 한 대당 시간당 수익 증가
- 택시 수요 감소 시 다른 수익원으로 위험 분산
- 전체 차량 수 감소 가능성
- 자율주행차의 높은 초기 투자비 회수에 유리

### 주요 맹점

- 택시와 택배의 차량 구조 및 적재 공간 차이
- 업무 전환을 위한 청소·적재·하역 시간
- 택배 완료 후 다음 택시 피크 지역으로 복귀해야 하는 문제
- 공차 이동 비용
- 충전·정비 시간 부족 가능성
- 택시와 물류 수요 피크가 동시에 발생할 가능성
- 승객 운송과 화물 운송의 보험·규제 차이
- 단순 매출이 아니라 실제 순이익을 비교해야 한다는 점

따라서 업무 전환의 가치는 다음과 같이 판단해야 한다.

```text
업무별 기대 순이익
= 예상 매출
- 운행 비용
- 공차 이동 비용
- 전환 비용
- 충전 비용
- 적재·하역 비용
- 다음 고수익 업무를 놓치는 기회비용
```

## 추가 서비스 아이디어: 이동식 휴식 차량

이동식 호텔은 장거리 숙박 차량이 아니라, 직장인이 점심시간처럼 짧은 시간 동안 호출해 이용하는 이동식 휴식 공간으로 정의했다.

사용자가 차량을 호출하면 자율주행차가 사용자를 태운 뒤 가까운 합법적 정차 공간으로 이동하고, 그곳에서 개인 휴식 공간을 제공한다.

```text
사용자 호출
→ 차량이 사용자 위치로 이동
→ 사용자 탑승
→ 주변 정차 가능 공간 탐색
→ 최적 공간으로 자율주행
→ 휴식 모드 전환
→ 예약 종료 후 다음 업무 수행
```

정차 공간은 단순히 가장 가까운 장소가 아니라 다음 조건을 함께 고려해 선택해야 한다.

- 주차·정차 가능 여부
- 현재 이용 가능성
- 이동시간
- 소음
- 안전성
- 프라이버시
- 주차 비용
- 이용 종료 후 다음 업무 지역으로의 접근성

이 서비스 역시 택시 수요가 낮은 점심시간의 유휴 차량을 다른 용도로 전환한다는 점에서 멀티서비스 자율주행차 아이디어와 연결된다.

## 이번 프로젝트의 핵심 분석 질문

이번 프로젝트에서는 우선 구현 가능성이 높은 택시와 택배 업무에 집중한다.

핵심 질문은 다음과 같다.

> 시간대·지역별로 택시 승객 수요가 낮을 때 택배 수요가 높게 나타나는가?

그리고 한 단계 더 나아가 다음을 확인한다.

> 택시 수요가 낮은 시간대에 유휴 자율주행차를 택배 차량으로 전환했을 때, 다음 택시 수요 피크 이전에 복귀할 수 있으며 전환 비용을 제외한 순이익이 대기 상태보다 높은가?

## 필요한 택배 데이터

택시 수요와 택배 수요의 상호보완성을 확인하려면 택배 데이터도 TLC와 유사하게 시간·지역·업무량·소요시간 단위로 구성되어야 한다.

### 최소 필요 컬럼

```text
package_id
request_datetime
pickup_datetime
delivery_datetime
pickup_zone_id
delivery_zone_id
delivery_distance
package_count
delivery_status
```

### 경제성 검증을 위한 추가 컬럼

```text
delivery_revenue
package_weight
package_volume
loading_time
unloading_time
delivery_deadline
vehicle_id
route_id
```

### TLC 데이터와의 대응 관계

| TLC 택시 데이터 | 택배 데이터 |
|---|---|
| `tpep_pickup_datetime` | `pickup_datetime` |
| `tpep_dropoff_datetime` | `delivery_datetime` |
| `PULocationID` | `pickup_zone_id` |
| `DOLocationID` | `delivery_zone_id` |
| `trip_distance` | `delivery_distance` |
| 택시 운행 건수 | 택배 건수 |
| `fare_amount`, `total_amount` | `delivery_revenue` |
| `passenger_count` | `package_count`, 물품 무게·부피 |

단, 택시 1건과 택배 1건을 그대로 비교하면 안 된다.

```text
택시 1건
= 차량 한 대가 한 번의 승객 운행 수행

택배 여러 건
= 차량 한 대가 한 번에 묶어서 운송 가능
```

따라서 최종 비교 단위는 단순 요청 건수보다 필요한 차량 수가 되어야 한다.

## 데이터 확보 전략

뉴욕에는 TLC처럼 개별 택배의 픽업·배송·거리·요금을 모두 공개한 Trip Record 데이터가 거의 없다.

따라서 다음 두 데이터를 결합하기로 했다.

### 1. LockerNYC Reservations

LockerNYC는 뉴욕시가 운영하는 공공 택배 보관함 서비스이다.

이 데이터는 뉴욕에서 실제로 발생한 보관함 예약·이용 기록을 제공하므로 다음 정보를 추정하는 데 사용한다.

- 뉴욕 지역별 택배 수요
- 시간대별 택배 활동량
- 보관함별 이용량
- 특정 지역에서 택배 수요가 집중되는 시점

LockerNYC는 뉴욕 전체 택배가 아니라 보관함을 이용한 일부 물량이므로 절대적인 전체 택배량이 아니라 지역·시간대별 상대 수요지수로 사용한다.

### 2. Amazon Last Mile Routing Challenge Dataset

Amazon Last Mile 데이터는 실제 미국 라스트마일 배송 경로를 익명화한 데이터이다.

이 데이터는 다음 값을 계산하는 데 사용한다.

- 경로별 배송 정류장 수
- 정류장별 택배 개수
- 정류장 간 이동시간
- 정류장별 서비스 시간
- 배송 시간창
- 차량 한 대의 시간당 택배 처리량

Amazon 데이터는 위치가 익명화되어 있어 뉴욕 Taxi Zone과 직접 조인하지 않는다.

```text
LockerNYC
→ 뉴욕의 지역·시간대별 택배 수요 분포

Amazon Last Mile
→ 차량 한 대의 배송 처리 구조와 처리량
```

즉, 두 데이터를 동일한 키로 직접 조인하는 것이 아니라 각 데이터에서 필요한 지표를 추출한 뒤 결합한다.

## LockerNYC와 Amazon 데이터 결합 방식

전체 처리 흐름은 다음과 같다.

```text
LockerNYC
→ 시간대·지역별 택배 건수 집계

Amazon Last Mile
→ 경로별 택배 수·정류장 수·운행시간 계산
→ 차량 1대의 시간당 평균 처리량 계산

두 결과 결합
→ 시간대·지역별 필요한 택배 차량 수 추정

TLC와 비교
→ 택시 저수요 시간에 택배 업무 전환 가능성 판단
```

### LockerNYC 수요 집계

```text
locker_id
date
day_of_week
hour
parcel_count
```

택배 수요지수는 다음과 같이 계산한다.

```text
택배 수요지수
= 해당 시간대의 LockerNYC 이용 건수
÷ 하루 중 최대 시간대 이용 건수
```

### Amazon 처리량 계산

```text
route_package_count
route_stop_count
route_duration
packages_per_hour
stops_per_hour
average_service_time
```

예를 들어 차량 한 대가 8시간 동안 180개의 택배를 처리했다면 시간당 처리량은 22.5개이다.

다만 배송 밴과 자율주행 택시의 적재능력이 다르므로 보정계수를 적용한다.

```text
자율주행 택시 시간당 처리량
= Amazon 배송 밴 시간당 처리량
× 적재 보정계수
```

### 필요한 택배 차량 수 계산

```text
필요 택배 차량 수
= 시간대별 예상 택배 건수
÷ 자율주행 택시 1대의 시간당 처리량
```

## TLC와의 비교 방법

TLC 데이터는 시간·요일·Taxi Zone별로 집계한다.

```text
시간대별 택시 수요
= 해당 시간대의 택시 승차 건수
```

택시 수요지수는 다음과 같이 계산한다.

```text
택시 수요지수
= 해당 시간대의 택시 승차 건수
÷ 하루 중 최대 시간대 승차 건수
```

두 수요지수를 비교해 다음과 같은 조건을 만족하는 시간대를 상호보완 후보로 분류한다.

```text
택시 수요지수 < 0.5
AND
택배 수요지수 > 0.7
```

예시:

| 시간대 | 택시 수요지수 | 택배 수요지수 | 판단 |
|---:|---:|---:|---|
| 08시 | 0.95 | 0.25 | 택시 유지 |
| 11시 | 0.48 | 0.72 | 일부 택배 전환 |
| 13시 | 0.35 | 1.00 | 택배 전환 유망 |
| 18시 | 1.00 | 0.40 | 택시 복귀 |

전체 뉴욕을 하나의 값으로 비교하지 않고, 시간대와 Taxi Zone별로 분석해야 한다.

```text
요일 × 시간대 × Taxi Zone
```

택시 수요가 낮은 지역과 택배 수요가 높은 지역이 서로 멀리 떨어져 있으면 공차 이동 비용이 커지기 때문이다.

## Prototype 기능

프로토타입은 다음 결과를 제공하는 대시보드 또는 Notebook 형태로 구현한다.

1. 시간대별 택시 수요 그래프
2. 시간대별 택배 수요 그래프
3. Taxi Zone별 택시·택배 수요 비교
4. 택시 저수요·택배 고수요 시간대 탐지
5. Amazon 데이터를 이용한 차량 한 대의 택배 처리량 추정
6. 시간대·지역별 필요한 택배 차량 수 계산
7. 택시 유지와 택배 전환 추천

예시 출력:

```text
요일: 월요일
시간: 13시
Taxi Zone: 161

택시 수요지수: 0.34
택배 수요지수: 0.91
예상 택배 물량: 80개
차량 1대 시간당 처리량: 8개
필요 택배 차량: 10대

추천 결과: 일부 차량을 택배 업무로 전환
```

## 분석 단계

```text
1. TLC 데이터 정제
2. TLC 시간대·지역별 택시 수요 집계
3. LockerNYC 데이터 정제
4. Locker 위치를 Taxi Zone으로 공간 조인
5. LockerNYC 시간대·지역별 택배 수요 집계
6. Amazon 경로·정류장·물품 데이터 정제
7. Amazon 경로별 시간당 처리량 계산
8. 자율주행 택시 적재 보정계수 적용
9. 택시·택배 수요지수 결합
10. 상호보완 후보 시간대 탐지
11. 필요 택배 차량 수 계산
12. 결과 시각화 및 업무 전환 추천
```

## 분석 시 확인할 지표

- 시간대별 택시·택배 수요 상관계수
- Taxi Zone별 택시·택배 수요 상관계수
- 택시 저수요·택배 고수요 시간대 비율
- 시간대별 예상 택배 필요 차량 수
- 택배 업무 후 다음 택시 피크 이전 복귀 가능률
- 공차 이동거리
- 택시 대기 대비 택배 전환 예상 순이익

음의 상관관계가 나타나면 택시와 택배 수요가 상호보완적일 가능성이 있다. 하지만 상관관계만으로 실제 업무 전환이 가능하다고 결론 내리면 안 되며, 위치·처리시간·적재량·복귀 가능성·비용을 함께 검토해야 한다.

## 데이터 한계

### LockerNYC의 한계

- 뉴욕 전체 택배 수요가 아니라 공공 보관함 이용 기록이다.
- 택배 출발지와 실제 배송 경로가 없다.
- 배송 차량 ID와 배송 수익이 없다.
- 실시간 택배 시장 전체를 대표하지 않는다.

### Amazon 데이터의 한계

- 위치가 익명화되어 있어 뉴욕 지역과 직접 연결할 수 없다.
- 실제 배송 수익이 없다.
- 자율주행 택시보다 큰 배송 밴을 기준으로 한다.
- 적재능력 차이를 보정계수로 가정해야 한다.

### TLC 데이터의 한계

- 차량 ID가 제한적이어서 실제 유휴 차량 수를 직접 계산하기 어렵다.
- 택시 운행 건수만으로 전체 가용 차량 수를 정확히 알 수 없다.
- 승객 수와 운행 건수에 누락·이상값이 존재할 수 있다.

따라서 이번 프로토타입은 실제 사업성을 확정하는 모델이 아니라, 공개 데이터를 활용해 택시와 택배 수요의 시간적·공간적 상호보완 가능성을 검증하는 초기 모델로 정의한다.

## 최종 Data Product

### 제품명

**Autonomous Vehicle Multi-Service Optimizer**

### 사용자

자율주행차 플릿 운영 관리자

### 해결하려는 문제

택시 수요가 낮은 시간에 차량이 대기하면서 가용시간과 수익성이 낭비된다.

### 입력 데이터

- TLC 택시 운행 데이터
- LockerNYC 택배 보관함 이용 데이터
- Amazon Last Mile 배송 처리 데이터
- 차량 위치·배터리·적재용량 합성 데이터
- 업무 전환 비용과 운행 비용 설정값

### 출력 결과

- 시간대·지역별 택시 수요
- 시간대·지역별 택배 수요
- 차량 한 대의 택배 처리 가능 물량
- 택배 업무에 필요한 예상 차량 수
- 택시 유지 또는 택배 전환 추천
- 다음 택시 피크 이전 복귀 가능 여부

### 기대 효과

- 자율주행차 유휴시간 감소
- 차량 가용시간당 평균 수익 향상
- 택시 수요 변화에 대한 유연한 대응
- 택시와 택배 차량의 통합 운영 가능성 검증
- 자율주행차를 단일 목적 차량이 아닌 이동 가능한 경제활동 플랫폼으로 확장

## 최종 정리

이번 아이디어의 핵심은 택시 수요가 낮다고 무조건 택배 업무로 전환하는 것이 아니다.

```text
택시 수요가 낮은가?
→ 같은 지역에 택배 수요가 존재하는가?
→ 차량 적재능력으로 처리 가능한가?
→ 다음 택시 피크 전에 복귀 가능한가?
→ 전환 비용을 제외해도 순이익이 발생하는가?
```

위 조건을 모두 만족할 때만 자율주행차의 업무를 택시에서 택배로 전환하도록 추천하는 것이 이번 Data Product의 최종 목표이다.
