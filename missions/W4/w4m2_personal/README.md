## 데이터에서 확인된 문제

### 1. `passenger_count` 결측치가 많음

전체 데이터는 3,399,866건이지만 `passenger_count`의 유효값은 2,376,549건이다.

```text
결측 건수 = 3,399,866 - 2,376,549
          = 1,023,317건
```

`passenger_count`는 평균 운행시간과 평균 운행거리 계산에 직접 사용하지 않으므로 필수 정제 대상에서는 제외한다. 다만 결측치가 많다는 점은 데이터 품질 이슈로 기록한다.

### 2. 운행거리 이상치가 존재함

```text
최소 운행거리: 0.0 mile
최대 운행거리: 328,522.2 mile
평균 운행거리: 약 6.24 mile
표준편차: 약 633.65
```

최대값 328,522.2mile은 현실적인 택시 운행거리로 보기 어렵다. 또한 표준편차가 평균보다 지나치게 큰 것으로 보아 극단적인 이상치가 포함되어 있다.

따라서 운행거리 정제 기준을 다음과 같이 설정한다.

```text
0 < trip_distance <= 100 miles
```

### 3. 비정상적인 승·하차 시각이 존재함

하차 시각이 승차 시각보다 빠르거나 같은 데이터가 40,598건 존재한다.

```text
Dropoff <= Pickup: 40,598건
```

이 데이터는 운행시간이 0 또는 음수가 되므로 다음 조건을 만족하는 데이터만 사용한다.

```text
tpep_dropoff_datetime > tpep_pickup_datetime
```

### 4. 0 이하 운행거리 데이터가 존재함

운행거리가 0 이하인 데이터가 123,250건 존재한다.

```text
trip_distance <= 0: 123,250건
```

이 값들은 평균 운행거리 계산을 왜곡할 수 있으므로 제거한다.

### 5. 운임 컬럼에 음수 및 극단값이 존재함

```text
fare_amount 최소값: -2,084.1
fare_amount 최대값: 2,084.1

total_amount 최소값: -2,088.85
total_amount 최대값: 2,088.85
```

음수 운임은 취소 또는 환불 기록일 가능성이 있으나, 현재 데이터만으로 정확한 원인을 확인할 수 없다. 이번 과제의 핵심 분석 대상은 운행시간과 운행거리이므로 운임 컬럼은 별도 정제하지 않고 데이터 품질 이슈로만 기록한다.

### 6. 파일명과 실제 데이터 기간이 정확히 일치하지 않음

`yellow_tripdata_2026-02.parquet` 파일의 실제 승차·하차 시각 범위는 다음과 같다.

```text
최소 승차 시각: 2026-01-31 23:31:23
최대 승차 시각: 2026-03-01 00:51:48
최소 하차 시각: 2026-01-31 23:36:52
최대 하차 시각: 2026-03-01 23:11:29
```

파일명은 2026년 2월 데이터지만 일부 1월 말과 3월 초 데이터가 포함되어 있다. 따라서 분석 기간을 정확히 2026년 2월로 제한하기 위해 다음 조건을 적용한다.

```text
2026-02-01 00:00:00 <= pickup_datetime < 2026-03-01 00:00:00
```

## 정제 기준

| 항목 | 정제 기준 |
|---|---|
| 승차 시각 | Null 제거 |
| 하차 시각 | Null 제거 |
| 운행거리 | Null 제거 |
| 분석 기간 | 2026-02-01 이상, 2026-03-01 미만 |
| 시간 순서 | 하차 시각 > 승차 시각 |
| 운행시간 | 1분 이상, 300분 이하 |
| 운행거리 | 0마일 초과, 100마일 이하 |
| 승객 수 | 결측 허용 |
| 운임 | 별도 정제하지 않고 품질 이슈로 기록 |
