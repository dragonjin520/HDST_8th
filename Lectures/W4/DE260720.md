## hadoop (mapreduce) VS spark

### 모든 앱은 초기 목적이 있다.
Spark의 최신 모델은 뭐든지 가능한 만능툴이 되었다.
평소에는 비슷해보이지만 대규모의 시스템에서는 한계가 들어난다.

### hadoop mapreduce의 이유
대규모 데이터의 분석에 의미가 있고 빠르게 결과를 내야할때
저렴한 하드웨어로 병렬 분석

#### batch의 단점
데이터 수집해서 분석하고 론칭, 피드백한다 -> 늦은 데이터
항상 늦게 대응하게 된다.


### spark의 이유
더 큰 대규모 데이터의 분석이 필요하다
분석을 기다리기 싫다. 바로바로 스트리밍 데이터로 분석하고 싶다.
    예시) 오늘 어떤 고객이 유튜브를 몇시간 시청했고 어떤 영상을 시청했는지. 오늘/이번주/이번달
    예시) * 더 중요한 데이터? 
            - 월드컵 시즌 시작 전 : 사람들의 기존 축구 시청 기록
            - 월드컵 시즌 시작 후 : 월드컵 기간동안 시청 기록
        * 즉 지금당장 생기는 데이터를 가지고 분석하고 싶다.

평소 환경과 급격하게 바뀌어서 고객들의 패턴이 바뀌는 것도 실시간으로 반영하고 싶다.

## Spark
- multi-language engine
    * high level APIs : java, scala, pandas 등등
- downsampling 단위 증가
    * 다른 앱에서는 십만에서 백이라면, Spark에서는 십만에서 만
- 실시간 데이터 처리 가능
- RDDs : 데이터가 날라갔을때 대처 방법
        * mapreduce와 달리 spark는 여러 단계를 거치기 때문에 오류가 발생했다고 해서 처음부터 다시 하기 비용이 너무 많이 든다.

- Cluster manager type
    - standalone
    - Yarn : hadoop
    - kubernets

### 구성
* Driver
    - worker node에 있는 parallel task를 실행한다.
    - worker node와 통신 필수
        - driver와 worker node의 통신이 끊어지면 job 날라간다.
        - master node와 연결한다고 끝이 아님.
        - AM (app master)와 비슷
* Executoer
    - task computation


### hadoop과 비교
Hadoop : executoer입장에서 mapper.py를 불러와서 데이터로 map 실행하고 꺼짐. reducer.py 불러와서 데이터 넘겨받아서 실행하고 꺼짐
Spark : Driver가 복잡한 모든 단계를 관리하면서 context를 이해해야함

### Client vs Cluster
Cluster
* Driver와 worker node가 같이 있음

Client
* Driver가 worker와 다른 곳(client와 같이)에 있음
    - Driver와 worker가 끊어지면 안되는데 왜 분리?
        - 오래걸리는 작업을 모니터링하면서 실시간으로 바꾸고 싶어
        - 안정화 전, 완벽한 개발 전에 고치는 단계에서 사용.
