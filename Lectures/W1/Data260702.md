## 1. 데이터 모델링 개요

데이터 모델링은 **데이터베이스에 저장할 데이터를 어떤 테이블로 나누고, 어떤 관계로 연결할지 설계하는 과정**이다.

쉽게 말하면 다음 질문에 답하는 과정이다.

* 어떤 데이터를 남길 것인가?
* 어떤 대상을 관리할 것인가?
* 각 대상은 어떤 정보를 가져야 하는가?
* 대상들은 서로 어떻게 연결되는가?

데이터 모델링의 핵심은 **무엇이 데이터로 남아야 하는지 골라내는 것**이다.

---

## 2. 모델링을 먼저 하는 이유

모델링을 먼저 하는 이유는 시스템을 더 쉽게 유지보수하고 수정하기 위해서이다.

데이터베이스는 한번 만들어지고 데이터가 쌓이면 구조를 바꾸기 어렵다.
특히 운영 중인 시스템에서 스키마를 변경하거나 데이터를 옮기는 **데이터 마이그레이션**은 큰 비용이 든다.

따라서 초기에 모델링을 통해 다음을 미리 확인해야 한다.

* 관리해야 할 데이터가 무엇인지
* 불필요한 데이터가 포함되어 있지는 않은지
* 관계가 잘못 설정되어 있지는 않은지
* 나중에 변경되기 어려운 구조는 아닌지
* 요구사항 중 불확실한 부분은 없는지

---

## 3. 데이터 모델링의 3대 개념

데이터 모델링의 핵심 개념은 다음 세 가지이다.

| 개념  | 영어           | 의미              |
| --- | ------------ | --------------- |
| 엔티티 | Entity       | 무엇을 관리할 것인가     |
| 속성  | Attribute    | 어떤 정보를 저장할 것인가  |
| 관계  | Relationship | 엔티티들이 어떻게 연결되는가 |

예를 들어 카페 주문 시스템을 생각하면 다음과 같이 볼 수 있다.

* 엔티티: 고객, 주문, 메뉴
* 속성: 고객ID, 고객이름, 주문ID, 주문수량, 메뉴ID, 가격
* 관계: 고객은 여러 주문을 할 수 있다

## 4. 개념 모델, 논리 모델, 물리 모델

데이터 모델링은 보통 다음 세 단계로 구체화된다.

## 4.1 개념 모델

개념 모델은 전체적인 큰 그림을 그리는 단계이다.

이 단계에서는 기술적인 구현보다 **무엇을 다룰지 합의하는 것**이 중요하다.

특징은 다음과 같다.

* 기술적인 세부사항은 무시한다.
* 핵심 엔티티와 관계를 찾는다.
* 이해관계자와 업무 범위를 합의한다.
* 시스템이 관리해야 할 대상을 정리한다.

## 4.2 논리 모델

논리 모델은 개념 모델을 더 상세하게 정리하는 단계이다.

이 단계에서는 엔티티의 속성, 관계, 규칙을 명확히 표현한다.

특징은 다음과 같다.

* 속성을 정의한다.
* 관계를 구체화한다.
* 식별자와 외래키 구조를 정리한다.
* DBMS에 독립적으로 정확한 구조를 표현한다.
* 업무 규칙을 데이터 구조에 반영한다.

## 4.3 물리 모델

물리 모델은 실제 DBMS에 맞춰 구현하는 단계이다.

예를 들어 MySQL, Oracle, SQL Server 등 특정 DBMS에 맞춰 테이블, 인덱스, 제약조건 등을 설계한다.

특징은 다음과 같다.

* 실제 테이블을 설계한다.
* 인덱스를 고려한다.
* 제약조건을 설정한다.
* 성능을 고려한다.
* 실제로 빠르게 동작하는 DB를 만드는 것이 목표이다.

성능 문제가 발생했을 때는 일반적으로 다음 순서로 접근한다.

```text
인덱스 튜닝 → 쿼리 수정 → 스키마 수정
```

스키마 수정은 영향 범위가 크기 때문에 가능한 뒤쪽 단계에서 고려한다.

## 5. 전통 기업과 IT 기업의 설계 관점 차이

전통적인 기업은 상대적으로 **데이터 중심 설계**를 중요하게 보는 경우가 많다.
반면 IT 기업은 **애플리케이션 중심 설계**를 많이 고려한다.

그 이유는 IT 서비스의 요구사항이 자주 바뀌기 때문이다.

요구사항이 자주 바뀌는 환경에서는 데이터 구조를 처음부터 완벽하게 고정하기보다, 애플리케이션의 변화에 맞춰 유연하게 설계하는 것이 중요하다.

최근 개발 동향에서는 DDD도 자주 언급된다.

DDD는 **Domain Driven Design**, 즉 도메인 주도 설계이다.
비즈니스 도메인을 중심으로 소프트웨어 구조를 설계하는 방식이다.

## 6. 객체지향 개발과 데이터 모델링

객체지향 개발에서 객체는 **상태와 행동**을 가진다.

* 상태: 객체가 가지고 있는 데이터
* 행동: 객체가 수행하는 기능

객체지향 개발에서는 테이블을 나누듯이 객체를 나누고, 각 객체가 책임과 역할을 가지도록 설계한다.

중요한 점은 객체지향에서는 단순히 상태보다 **행동이 더 중요하다**는 것이다.
객체는 데이터를 담는 그릇이 아니라, 역할과 책임을 수행하는 단위이다.

## 7. DB의 종류

DB는 크게 관계형 DB와 NoSQL로 나눌 수 있다.

## 7.1 관계형 DB

관계형 DB는 테이블 기반으로 데이터를 저장한다.

예시는 다음과 같다.

* MySQL
* Oracle
* Microsoft SQL Server

관계형 DB는 정해진 스키마, 관계, 제약조건을 기반으로 데이터를 안정적으로 관리하는 데 강점이 있다.

## 7.2 NoSQL

NoSQL은 관계형 DB보다 유연한 데이터 구조를 지원한다.

예시는 다음과 같다.

* MongoDB

NoSQL은 비정형 데이터, 빠른 확장, 유연한 구조 변경이 필요한 상황에서 사용될 수 있다.

## 8. 엔티티

엔티티는 업무에서 관리해야 하는 대상이다.

엔티티를 판단할 때는 다음 질문을 해볼 수 있다.

```text
이 대상을 추적할 필요가 있는가?
```

관리하거나 추적할 필요가 있다면 엔티티 후보가 될 수 있다.

엔티티는 인스턴스의 집합이다.
즉, 하나만 존재하는 것이 아니라 여러 개의 인스턴스가 모일 수 있어야 한다.

엔티티로 보기 위한 기준은 다음 세 가지이다.

1. 업무에 필요한가?
2. 여러 개 모일 수 있는가?
3. 식별이 가능한가?

---

## 9. 엔티티의 종류

엔티티의 종류는 여러 가지가 있지만, 중요한 것은 크게 두 가지이다.

## 9.1 개체 엔티티

개체 엔티티는 실제로 존재하는 대상이다.

예시는 다음과 같다.

* 회원
* 상품
* 고객
* 메뉴

개체 엔티티는 행위보다 먼저 존재하는 대상이다.

## 9.2 행위 엔티티

행위 엔티티는 사건이나 행동을 나타낸다.

예시는 다음과 같다.

* 주문
* 결제
* 예약
* 배송

행위 엔티티는 여러 개체 엔티티가 만나서 생성된다.
예를 들어 주문은 고객과 상품이 연결되면서 발생하는 행위이다.

행위 엔티티에서는 시간 정보가 중요해지는 경우가 많다.

---

## 10. 속성

속성은 엔티티 안에서 관리하고 싶은 정보이다.

예를 들어 고객 엔티티에는 다음과 같은 속성이 있을 수 있다.

* 고객ID
* 고객이름
* 전화번호
* 가입일

속성은 값들의 집합으로 볼 수 있다.

---

## 11. 좋은 속성의 조건

## 11.1 원자값

좋은 속성은 원자값을 가져야 한다.

원자값이란 더 이상 나누지 않아도 되는 최소 단위의 값이다.
다만 원자값의 기준은 업무 요구사항에 따라 달라질 수 있다.

예를 들어 주소를 하나의 문자열로 저장할 수도 있고, 시/군/구, 상세주소, 우편번호로 나눌 수도 있다.
어느 쪽이 맞는지는 업무에서 주소를 어떻게 사용하는지에 따라 달라진다.

일반적으로는 먼저 최소 단위로 쪼개고, 필요에 따라 합치는 방식이 좋다.

## 11.2 단일값

좋은 속성은 하나의 인스턴스에 대해 하나의 값만 가져야 한다.

예를 들어 한 고객이 여러 전화번호를 가질 수 있다면, 전화번호를 고객 테이블의 한 속성에 여러 개 넣는 것은 좋지 않다.

이 경우 전화번호를 별도 엔티티로 분리하는 것이 좋다.

```text
고객 1 : N 고객전화번호
```

반복되는 값은 별도 엔티티로 분리하는 것을 고려해야 한다.

---

## 12. 속성 검증 4단계

속성을 검증할 때는 다음 네 단계를 고려할 수 있다.

## 12.1 최소 단위 분할

속성이 업무상 필요한 최소 단위로 나누어져 있는지 확인한다.

## 12.2 유일값 검증

하나의 인스턴스에 대해 해당 속성이 하나의 값만 가지는지 확인한다.

## 12.3 가공값 판정

다른 값으로 계산할 수 있는 값인지 확인한다.

예를 들어 총 주문금액은 주문수량과 상품가격으로 계산할 수 있다.
이런 값은 논리 모델에서는 생략할 수 있다.

다만 성능상의 이유로 물리 모델에서는 저장할 수도 있다.

## 12.4 관리수준 검토

그 값을 실제로 관리할 필요가 있는지 확인한다.

재현 가능한 값은 논리 모델에서 생략하는 것이 일반적이다.
하지만 매번 계산하는 비용이 크다면 캐시 DB나 별도 컬럼을 둘 수도 있다.

예를 들어 `count` 값은 매번 세는 방식과 따로 저장하는 방식의 성능 차이가 클 수 있다.

---

## 13. 필수 속성과 선택 속성

속성은 필수 속성과 선택 속성으로 나눌 수 있다.

* 필수 속성: 반드시 값이 있어야 하는 속성
* 선택 속성: 값이 없어도 되는 속성

예를 들어 회원가입 시 이름은 필수일 수 있지만, 추천인 코드는 선택일 수 있다.

중요한 점은 **필수와 선택을 나누는 기준도 모델링의 일부**라는 것이다.

---

## 14. 관계

관계는 엔티티와 엔티티가 어떻게 연결되는지를 나타낸다.

관계는 참조 무결성을 지켜주는 핵심 장치이다.

예를 들어 존재하지 않는 고객의 주문은 있을 수 없다.

```text
고객 1 : N 주문
```

이 관계에서 주문은 고객을 참조한다.
고객 테이블의 PK가 주문 테이블의 FK로 들어가면서 연결이 만들어진다.

다만 참조 무결성을 매번 DB 제약조건으로 검사하면 성능 비용이 발생할 수 있다.
외래키 제약조건은 고객 테이블의 PK 인덱스를 확인해야 하므로 성능에 영향을 줄 수 있다.

그래서 실무에서는 상황에 따라 외래키 제약조건을 사용하지 않고 애플리케이션에서 관리하는 경우도 있다.

---

## 15. 관계의 3가지 표현

관계는 다음 세 가지로 표현한다.

## 15.1 관계명

관계가 어떤 의미인지 나타내는 이름이다.

예를 들어 다음과 같이 표현할 수 있다.

```text
고객은 주문한다.
주문은 상품을 포함한다.
```

## 15.2 기수성

기수성은 하나의 인스턴스가 상대 엔티티의 인스턴스 몇 개와 연결될 수 있는지를 나타낸다.

대표적인 기수성은 다음과 같다.

* 1:1
* 1:N
* M:N

## 15.3 선택성

선택성은 관계가 필수인지 선택인지를 나타낸다.

예를 들어 고객은 주문을 하지 않을 수도 있다.
이 경우 고객 입장에서 주문은 선택 관계이다.

반대로 주문은 반드시 고객이 있어야 한다.
이 경우 주문 입장에서 고객은 필수 관계이다.

기수성과 선택성은 방향에 따라 다르게 해석해야 한다.

---

## 16. 관계속성과 FK 상속

1:N 관계에서는 부모 엔티티의 PK가 자식 엔티티의 FK로 상속된다.

예를 들어 고객과 주문의 관계를 보면 다음과 같다.

```text
고객(고객ID) 1 : N 주문(주문ID, 고객ID)
```

여기서 `고객ID`는 고객 테이블에서는 PK이고, 주문 테이블에서는 FK이다.

설계 단계에 따라 표현은 다음과 같이 발전한다.

```text
엔터티 → 릴레이션 → 테이블
```

---

## 17. 식별 관계와 비식별 관계

FK가 자식 엔티티에서 어디에 위치하느냐에 따라 식별 관계와 비식별 관계로 나눌 수 있다.

## 17.1 식별 관계

식별 관계는 부모의 PK가 자식의 PK 일부가 되는 관계이다.

특징은 다음과 같다.

* FK가 자식 PK의 일부가 된다.
* 부모 없이 자식이 존재할 수 없다.
* 식별 종속이 있다.
* 까마귀발 표기에서는 실선으로 표현한다.

예를 들어 약개체는 식별 관계를 가지는 경우가 많다.

## 17.2 비식별 관계

비식별 관계는 부모의 PK가 자식의 일반 컬럼 FK로 들어가는 관계이다.

특징은 다음과 같다.

* FK가 자식의 일반 컬럼이다.
* 자식의 식별자는 독립적으로 존재한다.
* 식별 종속이 없다.
* 까마귀발 표기에서는 점선으로 표현한다.

대부분의 관계는 비식별 관계로 설계되는 경우가 많다.

---

## 18. 1:1 관계

1:1 관계는 드물게 발생한다.

1:1 관계가 보인다면 사실 같은 엔티티일 가능성도 있다.
따라서 굳이 나눌 이유가 있는지 확인해야 한다.

1:1로 나눌 수 있는 경우는 다음과 같다.

* 자주 사용하지 않는 큰 컬럼을 분리할 때
* 접근 권한이 다른 민감 정보를 분리할 때
* 성능상 자주 불러오는 데이터와 그렇지 않은 데이터를 나눌 때

예를 들어 회원 기본정보와 회원 민감정보를 분리할 수 있다.

```text
회원 1 : 1 회원민감정보
```

---

## 19. 1:N 관계

1:N 관계는 가장 일반적인 관계이다.

예시는 다음과 같다.

```text
고객 1 : N 주문
주문 1 : N 주문상세
상품 1 : N 주문상세
```

하나의 고객은 여러 주문을 할 수 있고, 하나의 주문은 여러 주문상세를 가질 수 있다.

---

## 20. M:N 관계

M:N 관계는 관계형 DB에서 직접 표현하기 어렵다.

따라서 교차 엔티티를 만들어 1:N 관계 두 개로 분해해야 한다.

예를 들어 주문과 상품은 M:N 관계처럼 보일 수 있다.

```text
주문 M : N 상품
```

하지만 관계형 DB에서는 다음처럼 풀어야 한다.

```text
주문 1 : N 주문상세 N : 1 상품
```

여기서 `주문상세`가 교차 엔티티이다.

---

## 21. M:N 관계를 교차 엔티티로 푸는 경우

M:N 관계는 다음과 같은 경우 교차 엔티티로 풀어야 한다.

1. 관계 자체에 속성이 있을 때
2. 관계가 자식 엔티티를 가질 때
3. 관계명만으로 의미를 명확히 붙이기 어려울 때

예를 들어 주문과 상품 사이에는 주문수량, 주문가격 같은 속성이 필요하다.
이 속성은 주문에도, 상품에도 완전히 속하기 애매하다.

따라서 `주문상세`라는 교차 엔티티를 만들어 관리한다.

---

## 22. 약개체

약개체는 부모 키와 합쳐야 구별되는 개체이다.

즉, 자기 자신의 속성만으로는 식별이 어렵고 부모 엔티티의 식별자가 함께 있어야 한다.

예를 들어 주문상세가 다음과 같은 식별자를 가질 수 있다.

```text
주문ID + 주문상세번호
```

이 경우 주문상세는 주문에 종속된 약개체로 볼 수 있다.

---

## 23. 슈퍼타입과 서브타입

같은 엔티티이지만 종류가 다른 경우 슈퍼타입과 서브타입으로 나눌 수 있다.

예를 들어 회원은 개인회원과 법인회원으로 나눌 수 있다.

```text
회원
├── 개인회원
└── 법인회원
```

공통 속성은 슈퍼타입에 둔다.
차이가 나는 속성은 서브타입에 둔다.

예를 들어 다음과 같이 나눌 수 있다.

* 회원: 회원ID, 가입일, 회원상태
* 개인회원: 이름, 생년월일
* 법인회원: 사업자등록번호, 법인명

주의할 점은 다음과 같다.

* 서브타입끼리는 겹치지 않게 한다.
* 모든 회원은 반드시 하나의 서브타입에 속하게 한다.
* 애매한 경우 기타 타입을 둘 수 있다.
* 슈퍼타입/서브타입은 부모/자식 관계와는 관점이 다르다.
* 차이가 없다면 굳이 나누지 않는다.

---

## 24. 데이터 시뮬레이션

데이터 시뮬레이션은 가장 저렴한 검증 방법이다.

모델을 설계한 뒤 샘플 데이터를 몇 줄 넣어보면 오류나 오해를 빠르게 발견할 수 있다.

장점은 다음과 같다.

* 적은 비용으로 검증할 수 있다.
* 의사소통 오류를 발견할 수 있다.
* 샘플 테스트 데이터로 재활용할 수 있다.
* 실제 업무 흐름에 맞는지 확인할 수 있다.

예를 들어 카페 주문 시스템을 설계했다면 실제 주문 데이터를 몇 줄 넣어본다.

```text
고객: C001, 김철수
메뉴: M001, 아메리카노, 4500원
주문: O001, C001
주문상세: O001, M001, 2잔
```

이렇게 넣어보면 주문수량이 어디에 있어야 하는지, 메뉴 가격과 주문 당시 가격을 어떻게 관리해야 하는지 같은 문제를 발견할 수 있다.

---

## 25. 미니 실습: 카페 주문 시스템

카페 주문 시스템의 엔티티 후보는 다음과 같다.

* 메뉴
* 주문
* 고객

각 엔티티의 속성 예시는 다음과 같다.

| 엔티티 | 속성         |
| --- | ---------- |
| 메뉴  | 메뉴ID, 가격   |
| 주문  | 주문ID, 주문수량 |
| 고객  | 고객ID, 고객이름 |

고객과 주문은 1:N 관계이다.

```text
고객 1 : N 주문
```

즉, 한 명의 고객은 여러 주문을 할 수 있다.

다만 실제 모델에서는 주문수량이 주문 엔티티에 직접 들어가기보다, 주문상세 엔티티에 들어가는 것이 더 자연스럽다.

예를 들어 하나의 주문에 여러 메뉴가 포함될 수 있기 때문이다.

```text
고객 1 : N 주문
주문 1 : N 주문상세
메뉴 1 : N 주문상세
```

---

## 26. 쇼핑몰 주문 시스템 ERD 예시

수업 내용을 바탕으로 쇼핑몰 주문 시스템을 까마귀발 표기 방식으로 간단히 표현하면 다음과 같다.

```text
고객 |───|< 배송지

고객 |───o< 주문 |───|< 결제 >o───| 결제수단

주문 |───|< 주문상세 >|───| 상품

주문 >o───o 배송지
```

## 26.1 해석

### 고객과 배송지

```text
고객 1 : N 배송지
```

한 명의 고객은 여러 배송지를 등록할 수 있다.
각 배송지는 특정 고객에게 속한다.

### 고객과 주문

```text
고객 1 : N 주문
```

한 명의 고객은 여러 주문을 할 수 있다.
주문은 특정 고객에 의해 생성된다.

### 주문과 결제

```text
주문 1 : N 결제
```

하나의 주문은 여러 결제로 나뉠 수 있다.
예를 들어 신용카드와 포인트를 함께 사용해 결제할 수 있다.

### 결제와 결제수단

```text
결제 N : 1 결제수단
```

각 결제는 하나의 결제수단을 사용한다.
결제수단은 여러 결제에서 사용될 수 있다.

### 주문과 주문상세

```text
주문 1 : N 주문상세
```

하나의 주문은 여러 상품을 포함할 수 있다.
따라서 주문과 상품 사이의 M:N 관계를 주문상세로 풀어준다.

### 주문상세와 상품

```text
주문상세 N : 1 상품
```

각 주문상세는 하나의 상품을 가리킨다.
상품은 여러 주문상세에 등장할 수 있다.

### 주문과 배송지

```text
주문 N : 1 배송지
```

하나의 주문은 하나의 배송지를 선택한다.
하나의 배송지는 여러 주문에서 사용될 수 있다.

다만 주문 당시 배송지 정보를 그대로 보존해야 한다면, 배송지ID만 참조하는 것이 아니라 주문 테이블에 수령인명, 주소, 연락처 등을 별도로 저장하는 방식도 고려할 수 있다.

## 27. 정리

이번 수업을 통해 데이터 모델링은 단순히 테이블을 나누는 작업이 아니라, 업무에서 관리해야 할 대상을 정의하고, 그 대상 사이의 관계를 명확히 하는 과정이라는 것을 알게 되었다.

특히 중요한 점은 다음과 같다.

* 엔티티는 관리해야 하는 대상이다.
* 속성은 엔티티 안에서 관리할 정보이다.
* 관계는 엔티티 사이의 연결이다.
* 개념 모델, 논리 모델, 물리 모델은 점점 구체화되는 과정이다.
* M:N 관계는 교차 엔티티로 풀어야 한다.
* 식별 관계와 비식별 관계는 FK가 자식 엔티티에서 어디에 위치하는지에 따라 달라진다.
* 데이터 시뮬레이션은 모델링 오류를 빠르게 찾을 수 있는 좋은 방법이다.
* 모델링의 목적은 유지보수와 변경이 쉬운 구조를 만드는 것이다.

데이터 모델링은 결국 “무엇을 데이터로 남길 것인가”를 판단하는 과정이며, 좋은 모델링은 이후 개발, 유지보수, 성능 개선의 기반이 된다.
