## 2. Trial and Error

Trial and Error는 최소한의 비용으로 최대한 많이 피드백을 받는 방식이다.

처음부터 완벽한 결과물을 만들려고 하기보다, 작은 단위로 시도하고 빠르게 검증하면서 개선하는 것이 중요하다.

데이터 엔지니어링에서도 마찬가지로 처음부터 완벽한 파이프라인이나 모델을 만들기보다, 작게 만들고 검증하면서 점진적으로 개선하는 방식이 필요하다.

## 3. 기술의 추상화와 데이터 직무

최근에는 기술의 추상화가 계속 일어나고 있다.

데이터 직무가 커지고 세분화되면서, 하나의 사람이 모든 것을 직접 구현하기보다 역할이 나누어지고 있다.  
이 과정에서 각 영역은 더 높은 수준의 추상화된 도구와 개념을 사용하게 된다.

예를 들어 데이터 직무 안에서도 다음과 같이 역할이 나뉠 수 있다.

- Data Analyst
- Data Architect
- Data Engineer
- Machine Learning Engineer
- Data Scientist

분업화가 진행될수록 각 직무는 자신의 역할과 책임을 명확히 이해해야 한다.

## 4. 데이터의 가치

Data는 단순한 정보가 아니라, 활용 방식에 따라 금전적 가치가 있는 정보로 바뀔 수 있다.

데이터 자체가 중요한 것이 아니라, 데이터를 통해 어떤 의사결정을 하고 어떤 행동을 할 수 있는지가 중요하다.

즉, 데이터는 다음 과정을 통해 가치가 생긴다.

```text
정보 수집 → 정리 → 분석 → 판단 → 행동 → 가치 창출
```
## 5. 문제 정의

데이터를 다루기 전에 가장 먼저 해야 할 일은 문제를 정확히 정의하는 것이다.

문제가 명확하지 않으면 어떤 데이터를 수집해야 하는지, 어떤 방식으로 처리해야 하는지, 어떤 결과를 만들어야 하는지 정하기 어렵다.

따라서 데이터 프로젝트에서는 먼저 다음 질문을 해야 한다.

- 해결하려는 문제는 무엇인가?
- 어떤 의사결정에 필요한 데이터인가?
- 이 데이터를 통해 어떤 행동을 할 수 있는가?
- 성공 기준은 무엇인가?

## 6. Action Data

데이터는 행동을 만들 수 있는지에 따라 구분할 수 있다.

### 6.1 Action Data

Action Data는 실제 행동을 하게 만드는 데이터이다.

예를 들어 다음과 같은 데이터는 Action Data가 될 수 있다.

재고 부족 알림
고객 이탈 가능성 예측
서버 장애 징후
매출 급감 지표
특정 상품의 수요 증가

이 데이터들은 단순히 보고 끝나는 것이 아니라, 실제 조치를 유도한다.

### 6.2 Action Data가 아닌 데이터

반대로 행동으로 이어지지 않는 데이터도 있다.

단순히 수집만 되고 활용되지 않는 데이터는 가치가 낮을 수 있다.

따라서 데이터 프로젝트에서는 “이 데이터가 어떤 행동으로 이어지는가?”를 계속 질문해야 한다.

## 7. DA와 DE의 차이
### 7.1 DA: Data Architecture

Data Architecture는 데이터 관리를 위한 효율적인 구조를 설계하는 역할이다.

목표는 데이터의 전체 구조를 설계하고, 데이터가 잘 관리될 수 있도록 청사진을 만드는 것이다.

- DA = Data Blueprint

주요 관심사는 다음과 같다.

데이터 구조
데이터 표준
데이터 모델
데이터 거버넌스
데이터 관리 체계
### 7.2 DE: Data Engineering

Data Engineering은 데이터를 실제로 수집, 처리, 저장, 제공하는 시스템과 파이프라인을 만드는 역할이다.

- DE = System + Pipeline

목표는 품질 좋은 데이터를 만들고, 실제로 활용할 수 있는 데이터로 제공하는 것이다.

주요 관심사는 다음과 같다.

데이터 수집
데이터 파이프라인
ETL/ELT
데이터 품질
데이터 저장소
자동화
모니터링

## 8. AI와 프로젝트를 진행하는 방법

AI와 프로젝트를 진행할 때 처음부터 전체 코드를 한 번에 만들어 달라고 하면 실패할 가능성이 높다.

한 번에 전체를 만들면 문제가 생겼을 때 어느 단계에서 잘못되었는지 추적하기 어렵고, 다시 이전 단계로 돌아가기도 어렵다.

따라서 AI와 협업할 때는 다음 방식이 좋다.

큰 문제를 작은 단계로 분해
→ 한 단계 구현
→ 리뷰
→ 피드백
→ 수정
→ 다음 단계 진행

중요한 원칙은 다음과 같다.

한 번에 통째로 만들라고 하지 않는다.
단계별로 작업을 나눈다.
각 단계가 끝날 때마다 리뷰한다.
AI의 답변을 그대로 믿지 않는다.
근거를 요구한다.
결과를 직접 검증한다.

AI는 도구로 활용해야 하며, 최종 판단은 사용자가 해야 한다.

## 9. Data Type

데이터 타입은 크게 세 가지로 구분할 수 있다.

### 9.1 Structured Data

Structured Data는 정해진 구조를 가진 데이터이다.

예시는 다음과 같다.

관계형 DB 테이블
엑셀 표
CSV 파일

행과 열이 명확하고, 스키마가 정해져 있는 경우가 많다.

### 9.2 Semi-Structured Data

Semi-Structured Data는 완전히 고정된 테이블 구조는 아니지만 일정한 형식을 가진 데이터이다.

예시는 다음과 같다.

JSON
XML
로그 데이터
### 9.3 Unstructured Data

Unstructured Data는 정해진 구조가 없는 데이터이다.

예시는 다음과 같다.

이미지
음성
영상
문서
자연어 텍스트

데이터 타입은 데이터 자체만으로 고정되는 것이 아니라, 어떻게 사용하고 어디에 저장하느냐에 따라 달라질 수 있다.

## 10. ETL

ETL은 데이터를 추출하고, 변환하고, 적재하는 과정이다.

Extract → Transform → Load
### 10.1 Extract

Extract는 데이터를 원천 시스템에서 가져오는 단계이다.

이 단계에서는 대규모 데이터를 안정적으로 가져오는 것이 중요하다.
다만 원천 시스템에 부하를 주지 않도록 주의해야 한다.

### 10.2 Transform

Transform은 가져온 데이터를 목적에 맞게 변환하는 단계이다.

예를 들어 다음과 같은 작업이 포함될 수 있다.

형식 변환
결측치 처리
중복 제거
데이터 정제
컬럼명 표준화
단위 변환

Transform 단계는 고성능 병렬 처리가 중요할 수 있다.

### 10.3 Load

Load는 변환된 데이터를 최종 저장소에 적재하는 단계이다.

예시는 다음과 같다.

Data Warehouse
Data Mart
Database
Data Lake
## 11. Staging Area

ETL 과정에서는 Extract와 Transform 단계를 구분하는 것이 중요하다.

이를 위해 Staging Area를 둘 수 있다.

Staging Area는 원천에서 추출한 데이터를 바로 최종 저장소에 넣기 전에 임시로 저장하고 변환하는 별도 공간이다.

Source → Staging Area → Target

Staging Area를 사용하는 이유는 다음과 같다.

원천 데이터 보존
변환 과정 분리
오류 추적 용이
재처리 가능
Extract와 Transform 역할 분리
파이프라인 안정성 향상

즉, Extract와 Transform을 한 번에 처리하지 말고, 중간 저장 공간을 두는 것이 좋다.

AI를 활용할 때도 한 번에 전체 결과물을 만들기보다, 작은 단계로 나누어 검증하고 피드백을 받으며 진행하는 것이 중요하다.
