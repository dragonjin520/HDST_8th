# 0713 Data Engineering

## Hadoop

Hadoop은 **저렴하고 불안정한 여러 대의 서버를 활용해 대규모 데이터를 저장하고 처리하기 위해 등장한 분산 처리 시스템**이다.

같은 목적을 가진 기술이라도 등장 배경과 해결하려는 문제가 다르므로, 단순한 사용법보다 왜 만들어졌는지와 장단점을 함께 이해하는 것이 중요하다.

---

## 1. Hadoop의 등장 배경

2000년대 초반 Google, Facebook, Yahoo와 같은 웹 서비스가 성장하면서 웹 데이터가 폭발적으로 증가했다.

당시에는 현재와 같은 클라우드 환경이 보편적이지 않았고, 서버 장애도 자주 발생했다. 더 크고 비싼 서버 한 대를 사용하는 방식만으로는 계속 증가하는 데이터를 감당하기 어려웠다.

이에 따라 Hadoop은 다음과 같은 방향으로 개발되었다.

> 큰 서버 한 대를 사용하는 대신, 저렴한 여러 대의 서버에 데이터를 분산 저장하고 병렬 처리한다.

### Hadoop이 해결하려는 문제

- 대규모 데이터를 저렴한 하드웨어에 저장
- 여러 서버에 데이터를 분산하여 처리
- 일부 서버에 장애가 발생해도 전체 작업을 계속 수행
- 복잡한 연산보다는 대량 데이터의 집계와 반복 처리에 집중

---

## 2. Hadoop의 계층 구조

Hadoop은 저장, 자원 관리, 데이터 처리를 각각 분리한다.

```text
HDFS      → YARN              → MapReduce
Storage     Resource Manager    Application
```

| 구성 요소 | 역할 |
|---|---|
| HDFS | 데이터를 여러 서버에 분산 저장 |
| YARN | 클러스터의 자원과 작업 실행 환경 관리 |
| MapReduce | 분산된 데이터를 병렬로 처리 |

각 역할이 분리되어 있기 때문에 구조가 단순하고, 각 계층을 독립적으로 관리하기 쉽다.

---

## 3. Master Node와 Slave Node

Hadoop 클러스터는 크게 Master Node와 Slave Node로 나뉜다.

| 구분 | 대표 구성 요소 | 역할 |
|---|---|---|
| Master Node | NameNode, ResourceManager | 전체 데이터 위치와 자원 관리 |
| Slave Node | DataNode, NodeManager | 실제 데이터 저장과 작업 수행 |

### NameNode

- 파일과 디렉터리의 메타데이터 관리
- 데이터 블록이 어느 DataNode에 저장되어 있는지 관리

### DataNode

- 실제 데이터 블록 저장
- NameNode의 요청에 따라 데이터 읽기 및 쓰기 수행

---

## 4. Task Parallelism과 Data Parallelism

### Task Parallelism

데이터를 작업이 수행되는 위치로 가져온 뒤 여러 작업을 병렬로 처리하는 방식이다.

```text
데이터 이동 → 작업 실행
```

### Data Parallelism

데이터가 저장된 위치에서 작업을 수행하는 방식이다.

```text
데이터가 있는 위치 → 작업 실행
```

Hadoop은 대규모 데이터를 이동시키는 비용이 크기 때문에 가능한 한 데이터가 저장된 노드에서 작업을 수행한다.

> Moving Computation is cheaper than Moving Data.

즉, 데이터를 작업 위치로 옮기기보다 작은 연산 코드를 데이터가 있는 위치로 보내는 것이 더 효율적이다.

각 데이터 조각은 독립적으로 처리할 수 있기 때문에 일부 작업이 실패하더라도 해당 작업만 다시 실행할 수 있다.

---

## 5. HDFS

HDFS는 대규모 데이터를 여러 서버에 분산 저장하는 파일 시스템이다.

### 주요 특징

#### Replication

동일한 데이터 블록을 여러 DataNode에 복제하여 저장한다. 일부 서버가 고장 나더라도 다른 복제본을 이용해 데이터를 계속 사용할 수 있다.

#### Large Data Sets

작은 파일을 자주 처리하기보다는 대용량 파일을 저장하고 순차적으로 읽는 작업에 적합하다.

#### Heterogeneous Hardware

성능과 사양이 서로 다른 여러 하드웨어를 하나의 클러스터로 구성할 수 있다.

#### Simple Coherency Model

HDFS는 기본적으로 WORM 방식을 따른다.

```text
Write Once, Read Many
```

한 번 저장한 데이터를 자주 수정하기보다 여러 번 읽는 환경에 적합하다. 따라서 로그, 이력 데이터, 원본 데이터 저장에 적합하다.

#### Data Locality

데이터를 연산 위치로 옮기지 않고, 데이터가 저장된 노드에서 연산을 수행한다. 이는 처리 시간뿐 아니라 네트워크 사용량과 전체 시스템 비용도 줄여준다.

---

## 6. YARN

YARN은 Hadoop 클러스터의 CPU, 메모리와 같은 자원을 관리하고 작업 실행을 조율한다.

| 구성 요소 | 역할 |
|---|---|
| ResourceManager | 클러스터 전체 자원 관리 및 ApplicationMaster 실행 자원 할당 |
| NodeManager | 각 노드의 자원과 Container 관리 |
| ApplicationMaster | 개별 애플리케이션 실행과 Container 상태 모니터링 |
| Container | 실제 작업이 실행되는 CPU·메모리 자원 단위 |

### 작업 흐름

```text
사용자 작업 요청
→ ResourceManager가 ApplicationMaster 실행
→ ApplicationMaster가 필요한 Container 요청
→ NodeManager가 Container 실행 및 관리
→ ApplicationMaster가 작업 상태 모니터링
```

---

## 7. MapReduce

MapReduce는 대규모 데이터를 여러 조각으로 나누어 병렬 처리한 뒤 결과를 합치는 방식이다.

### Map 단계

1. HDFS에서 데이터를 읽는다.
2. 데이터를 여러 Input Split으로 나눈다.
3. 각 Split을 Mapper가 병렬로 처리한다.
4. Key-Value 형태의 중간 결과를 생성한다.
5. 중간 결과를 로컬 디스크에 저장한다.

```text
Input Data
→ Split
→ Map
→ Intermediate Key-Value
```

### Shuffle and Sort 단계

Mapper의 중간 결과를 Key 기준으로 모으고 정렬한다.

```text
Intermediate Output
→ Shuffle
→ Sort
→ Group by Key
```

### Reduce 단계

같은 Key를 가진 값들을 집계하여 최종 결과를 생성한다.

예를 들어 단어 개수를 세는 작업은 다음과 같이 처리된다.

```text
Map:    (word, 1)
Shuffle: 같은 word끼리 모음
Reduce: word별 개수 합산
```

---

## 8. Hadoop의 핵심 의미

Hadoop은 실시간 처리보다 대량의 로그와 데이터를 저장한 뒤 사후에 일괄 처리하는 Batch Processing에 적합하다.

실시간으로 계속 집계하는 대신 로그를 저장해 두고 나중에 한꺼번에 처리하면 대규모 데이터에서 더 효율적일 수 있다.

### 핵심 정리

- Hadoop은 저렴한 여러 서버를 활용한 분산 시스템이다.
- HDFS는 데이터를 분산 저장하고 복제하여 장애에 대비한다.
- YARN은 클러스터의 자원과 작업 실행 환경을 관리한다.
- MapReduce는 데이터를 나누어 병렬 처리하고 결과를 합친다.
- 데이터를 이동하기보다 데이터가 있는 곳으로 연산을 보내는 것이 핵심이다.
- 일부 작업이 실패해도 해당 작업만 다시 실행할 수 있다.
- 실시간 처리보다는 대용량 로그의 사후 일괄 처리에 적합하다.
