## 오늘 주제 : 식별자와 키
논리 모델 -> 실제 DB 테이블로 내려간다.

식별자 : 유일하게 구분하는 속성 -> DB에서 키
    유일성, 최소성

슈퍼키 : 유일성 O, 최소성 X
후보키 : 유일성 O, 최소성 O
기본키 : 후보키 중 하나 선택
대체키 : 기본키 되지 못한 후보키들
    보통 UNIQUE 제약으로 살려둔다.
        - 자동으로 데이터 정합성이 지켜진다.
        - 매번 insert할때마다 검사하기 때문에 성능 하락 요인

외래키 : 다른 엔터티의 기본키 참조
    - 참조 무결성 중요
        - 테이블 복원할 때 외래키 덕분에 가능
    - 사용할때마다 FK가 기본키인지 검사
        - 인덱스가 있어서 빠르게 검색
    - 외래키 참조 무결성은 이론상 쓰는게 맞는데 현업에서 대체로 안쓰는 경향이 있음

## 자연키 VS 대리키
대리키 : 시스템이 인위적으로 만든 값 ex) auto_increment, UUID
자연키 : 내용상 식별자
    단점 : 깨질 수 있다.
        유일성 실패 : 설계상 완벽한 줄 알았는데 아니었다.
        너무 김/복잡함 : Join할 때마다 들고 다님
        값이 바뀜 : 나중에 변경가능성 있음
            PK가 바뀌면 FK가 전부 흔들림

인조키 : 유일성 보장, 편의성, 단순성, 변경X, 자손 많을떄
### UUID
    쓰는 이유
        - ID가 순차적으로 증가하면 정보가 세어나감. 
        - 분산시스템의 경우. 글로벌 서비스
        - 겹치지 않음
    단점 
        - 용량이 크다. 16byte
        - 인덱스가 커진다.
        - 정렬이 안되어있음. 시간 순서가 없음(최신 버전에서는 업데이트됨)

### 인조키 주의사항
    사용안하면 불필요한 인덱스 늘림
    자연키 UNIQUE 꼭 지켜라
    편하려고 쓰는 것

기본키 선정 기준 : NotNull, 불변성, 단순성, 대표성
